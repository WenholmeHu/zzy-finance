## 1. 背景与目标

需要对两张 Excel 表进行数据对账，并根据两张表中的字段完成关联、汇总和计算，最终输出一组对账结果。

本轮目标仅为让 AI 准确理解业务需求，不讨论实现方式，不编写正式对账程序。

## 3. 业务目标

先以订单粒度，将 `聚天下.xlsx` 与 `携程.xlsx` 基于唯一键完成关联；再按产品名称进行汇总，并补充输出双向订单差异检查结果。

本次统计范围有一个前置条件：

- 只统计 `携程.xlsx` -> `流水` 中 `出发时间` 在 `2026-03-01` 至 `2026-03-31`（含）范围内的记录
- 不属于上述时间范围的携程流水，不参与本次对账计算
- 双向订单差异检查也必须建立在“携程先完成当月过滤”这一前提之上

在满足上述范围条件后，系统需要输出两类结果：

### 3.1 主结果表

输出以下字段：

- 产品名称
- 核销人次
- 销售额
- 结算实付
- 采购金额
- 利润

### 3.2 订单差异检查结果

新增第二个结果区域，用于输出两个维度的订单不一致情况：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

差异判断规则为：

- 先按 `携程.xlsx` -> `流水` -> `出发时间` 过滤出当月记录
- 再对过滤后的第三方订单号集合与聚天下订单号集合做比较
- 差异结果按唯一订单号输出，不重复

## 4. 关联规则

已确认的关联规则为：

- `聚天下[订单号] == 携程[流水].[第三方单号]`

补充说明：

- 两张表的行顺序不是一一对应
- 但两张表中存在相同的这个号
- 因此应基于该号完成跨表关联，再生成最终输出
- `携程.xlsx` -> `流水` 中，同一个 `第三方单号` 可能出现多行
- 这些多行可视为同一订单下的多人或多份结算明细
- 在业务统计上，应将同一 `第三方单号` 下符合时间范围的 `结算价金额` 汇总后，再归属到对应的 `聚天下[产品内容]`

## 6. 计算规则

### 6.0 总体处理逻辑

处理逻辑可以明确为六步：

1. 先筛选 `携程.xlsx` -> `流水` 中 `出发时间` 在 `2026-03-01` 至 `2026-03-31`（含）的记录
2. 对同一个 `第三方单号` 对应的多条携程流水，按订单汇总 `结算价金额`
3. 再按关联键把 `聚天下.xlsx` 与筛选并归集后的携程订单关联起来
4. 按 `产品名称` 汇总，输出主结果表
5. 基于“聚天下订单号集合”和“当月第三方订单号集合”做双向差异比较
6. 输出主结果表以及两个订单差异明细结果

也就是说，最终结果不是单一的一张表，而是：

- 一个按产品名称汇总的主结果表
- 两个按唯一订单号展示的差异明细结果

### 6.1 产品名称

- 来源：`聚天下.xlsx` -> `订单列表` -> `产品内容`
- 作为最终结果表的汇总维度

### 6.2 核销人次

- 来源：`聚天下.xlsx` -> `订单列表` -> `实到人数`
- 规则：只对成功匹配且属于本月范围的订单，按产品名称分组后求和

### 6.3 销售额

- 来源：`携程.xlsx` -> `流水` -> `结算价金额`
- 规则：先筛选 `出发时间` 在 `2026-03-01` 至 `2026-03-31` 的流水
- 若同一个 `第三方单号` 存在多条流水，则先按订单汇总其 `结算价金额`
- 订单关联完成后，再按产品名称分组后求和

### 6.4 结算实付

- 来源：`携程.xlsx` -> `流水` -> `结算价金额`
- 规则：先筛选 `出发时间` 在 `2026-03-01` 至 `2026-03-31` 的流水
- 若同一个 `第三方单号` 存在多条流水，则先按订单汇总其 `结算价金额`
- 订单关联完成后，再按产品名称分组后求和
- 按当前需求，`销售额` 与 `结算实付` 的来源字段一致，因此两列汇总结果也会一致

### 6.5 采购金额

- 来源：`聚天下.xlsx` -> `订单列表` -> `采购金额`
- 规则：只对成功匹配且属于本月范围的订单，按产品名称分组后求和

### 6.6 利润

- 公式：`利润 = 结算实付合计 - 采购金额合计`

### 6.7 订单差异检查

- 差异判断必须建立在第三方先按当月过滤之后
- `聚天下有、第三方当月无`：
  使用 `聚天下.xlsx` -> `订单列表` -> `订单号`，判断其是否存在于“当月过滤后的第三方订单号集合”中
- `第三方有、聚天下当月无`：
  使用“当月过滤后的第三方订单号集合”，判断其是否存在于 `聚天下.xlsx` -> `订单列表` -> `订单号` 集合中
- 两个方向都按唯一订单号输出，不重复
- 差异结果只要求展示订单号本身，不要求补充更多字段

## 7. 建议的数据处理口径

以下内容属于建议口径，便于后续实现时统一规则：

- 先按携程 `出发时间` 过滤统计范围，再做订单关联
- 对同一 `第三方单号` 的多条携程流水，先按订单汇总金额
- 主结果表只统计成功匹配的订单
- 先按订单归集，再按产品名称汇总
- 以“一个产品名称”输出“一条结果记录”
- 双向差异检查基于“去重后的订单号集合”进行
- `聚天下有、第三方当月无` 与 `第三方有、聚天下当月无` 都按唯一订单号输出一次
- 金额字段按数值处理
- 输出时金额建议保留两位小数
