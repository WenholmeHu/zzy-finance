# 平台接入检查清单

文档用途：给后续 AI 开发和人工开发在“真正动代码前”快速过一遍，避免漏掉关键判断。

## 1. 样例文件检查

开始开发前，至少确认下面内容：

- 工作表名称
- 工作表名称是否固定
- 表头是否在第一行
- 是否存在多个工作表
- 订单关联字段
- 月份过滤字段
- 金额字段
- 是否存在退款 / 负数 / 冲正 / 调账
- 是否一单多行
- 差异比较使用哪个订单字段
- 订单号是否会出现 `12345.0`

## 2. 平台分类判断

先判断这个平台属于哪一类：

### A 类

- 输入结构不同
- 但结果结构兼容当前系统

### B 类

- 指标或计算更复杂
- 但仍然只有一个主结果表

### C 类

- 结果结构发生变化
- 例如多个主结果表、多工作表导出、多差异区

## 3. 接入前 6 个关键问题

### 3.1 聚天下用哪个字段匹配

- `订单号`
- `渠道订单号`
- 还是别的字段

### 3.2 平台需要几个工作表

- 单工作表
- 单工作表但表名动态
- 多工作表

### 3.3 平台按哪个日期字段过滤

- 出发时间
- 销售时间
- 核销时间
- 其他业务时间

### 3.4 平台差异表头是什么

- 内部侧表头
- 外部侧表头

### 3.5 主结果表列是什么

- 列标题
- 列顺序
- 指标键

### 3.6 利润与汇总规则是否仍然通用

如果不是，先停下来升级设计，不要直接写适配器。

## 4. 需要修改的代码位置

普通新增平台通常至少改：

- `app/platforms/<platform>_adapter.py`
- `app/platforms/registry.py`
- `app/platforms/report_definitions.py`
- `tests/platforms/test_<platform>_adapter.py`
- `tests/integration/test_<platform>_reconciliation.py`

如果平台文件只有一张表但表名动态，通常还会改：

- `app/application/reconciliation_service.py`
- `app/infrastructure/excel_reader.py`

如果页面或导出列结构变化，通常还会改：

- `tests/web/test_reconcile_route.py`
- `tests/web/test_export_route.py`

## 5. 一般不该先动的代码

下面这些文件不应作为“普通新增平台”的第一改动点：

- `app/domain/reconcile.py`
- `app/models/reconciliation.py`
- `app/web/templates/index.html`
- `app/infrastructure/excel_writer.py`

除非你已经确认这次是 B 类后段或 C 类扩展。

## 6. 最低测试清单

至少补齐：

- 适配器测试：
  - 缺工作表
  - 缺字段
  - 非当月过滤
  - 一单多行聚合
  - 指标映射

- 集成测试：
  - 完整对账
  - 匹配订单数
  - 汇总产品数
  - 至少一个关键产品指标
  - 差异订单结果

- Web / 导出测试：
  - 下拉框出现新平台
  - 主结果表头正确
  - 差异表头正确
  - 导出列顺序正确

## 7. 提交前自检

提交前至少确认：

- 新平台是否只把私有逻辑放在适配器 / 平台元数据 / 报表定义层
- 有没有把平台字段名硬编码到领域层
- 有没有遗漏差异表头
- 有没有遗漏真实样例集成测试
- 如果样例文件集发生切换，旧真实样例测试是否已改为跳过或同步更新
- README 和平台文档有没有同步更新
