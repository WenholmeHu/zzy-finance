## 2. 当前项目的扩展原则

当前项目已经按“平台适配器”方式实现了首个平台 `携程`。后续新增平台时，优先遵守以下原则：

1. 领域层只处理“统一后的标准数据”，不要把平台字段名直接写进领域层。
2. 每个平台单独一个适配器文件，不要把多平台逻辑堆在一个大 `if/else` 里。
3. 财务手动选择 `对账月份`，平台适配器负责按该月份过滤外部平台数据。
4. 前端只负责“选平台 + 传文件 + 看结果”，平台差异尽量收敛在后端适配层。
5. “文件接入差异”和“对账规则差异”是两类不同问题，不要混为一谈。
6. 当前业务默认输出不只有主结果表，还包含双向订单差异检查结果；后续扩平台时必须同时考虑这两部分。
7. 如果只是新增平台文件格式，一般不应该修改核心汇总算法；只有业务模型、计算公式或输出结构真的变了，才允许改领域层和结果模型。

## 4. 新平台接入前，先做 6 个判断

在真正写代码前，先判断新平台属于哪一类。

### 4.1 是否存在可用于关联聚天下订单的字段

当前系统的统一匹配关系是：

- `聚天下.order_no`
- `外部平台.external_order_no`

也就是说，新平台必须能提供一个字段，经过清洗后，可以映射成统一字段 `external_order_no`。

如果新平台没有任何字段能和聚天下订单号对应，不要直接开发，先停下来重新确认业务口径。

### 4.2 是否需要按“对账月份”过滤

当前已确认的项目规则是：

- 财务手动选择 `对账月份`
- 外部平台适配器必须按这个月份过滤

因此新增平台时，必须先识别：

- 该平台用于过滤月份的日期字段是什么
- 字段格式是否稳定
- 空值和异常值怎么处理

如果没有可用日期字段，也不要直接硬写逻辑，先确认业务上是否允许不做月份过滤。

### 4.3 新平台金额口径是否仍然是“单金额模型”

当前系统采用的是“单金额模型”：

- 统一模型里只有 `settlement_amount`
- 页面输出里的 `销售额` 和 `结算实付` 当前都取这个值

这适合当前携程接入方式。

如果新平台也能按这个口径接入，那么只需要新增适配器即可。

如果新平台明确存在两个不同金额字段，例如：

- `销售额`
- `结算实付`

且这两个字段业务上不能视为同一个值，那么不能只写适配器，必须先升级统一数据模型，再改领域层、导出层和测试。

这类改动属于“平台扩展 + 模型升级”，不要误判成普通新增平台。

### 4.4 新平台主结果表是否仍然是当前固定 6 列

当前业务默认主结果表输出列为：

- `产品名称`
- `核销人次`
- `销售额`
- `结算实付`
- `采购金额`
- `利润`

如果新平台最终也输出这 6 列，只是来源字段不同，那么主结果表大概率仍可复用当前结果模型。

如果新平台要求：

- 输出列不一样
- 列顺序不一样
- 多出额外指标
- 少掉某些指标
- 需要多个结果表
- 需要多工作表导出

那么这已经不是“只加适配器”的问题，而是“平台专属结果结构”问题。

这类情况下，不能继续把所有平台强塞进当前 `ProductSummaryRow`。

### 4.5 新平台订单差异检查结果是否仍然沿用当前结构

当前业务默认还要求输出两个差异结果区域：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

并且当前默认展示字段是：

- 前者只展示 `订单号`
- 后者只展示 `第三方单号`

如果新平台也沿用这个结构，那么差异结果层面可以继续复用当前设计思路。

如果新平台要求：

- 差异结果展示更多字段
- 差异结果不止两个方向
- 差异结果需要按别的键比较
- 差异结果需要额外汇总统计

那么这已经属于“结果结构变化”，不应只改适配器。

### 4.6 新平台计算公式是否与当前汇总公式一致

当前携程版本默认公式是：

- `核销人次 = 聚天下实到人数汇总`
- `销售额 = 外部平台金额汇总`
- `结算实付 = 外部平台金额汇总`
- `采购金额 = 聚天下采购金额汇总`
- `利润 = 结算实付 - 采购金额`

如果新平台只是字段名不同，但计算口径仍然一致，那么仍属于“普通平台扩展”。

如果新平台要求：

- 销售额和结算实付来自不同字段
- 利润公式不同
- 采购金额需要单独换算
- 汇总维度不是按 `产品内容`
- 需要额外的业务规则过滤

那么应该把它认定为“平台专属汇总规则”，而不是普通适配器问题。

## 5. 当前统一接口与数据模型

### 5.1 平台适配器接口

所有新平台都必须实现 `PlatformAdapter` 协议：

```python
class PlatformAdapter(ABC):
    platform_name: str
    worksheet_name: str

    @abstractmethod
    def parse(self, dataframe: pd.DataFrame, reconciliation_month: str) -> PlatformParseResult:
        raise NotImplementedError
```

当前含义如下：

- `platform_name`
  平台内部标识，例如 `ctrip`、`meituan`

- `worksheet_name`
  外部 Excel 中需要读取的工作表名

- `parse(...)`
  接收平台原始 `DataFrame` 和页面选择的 `对账月份`，返回统一解析结果

### 5.2 当前统一返回结构

当前平台适配器最终必须返回：

```python
PlatformParseResult(
    orders=[ExternalOrderAggregate(...)],
    filtered_out_of_month_row_count=...
)
```

其中核心对象是：

```python
ExternalOrderAggregate(
    external_order_no: str,
    settlement_amount: float,
    platform_name: str,
    source_row_count: int,
    business_date: date | None,
)
```

字段含义：

- `external_order_no`
  用于和聚天下 `order_no` 关联的外部订单号

- `settlement_amount`
  当前统一金额字段

- `platform_name`
  平台代号，便于调试与后续扩展

- `source_row_count`
  这个聚合订单由原始表多少行合并而来

- `business_date`
  该聚合订单对应的业务日期；当前主要用于保留调试信息

### 5.3 当前业务默认结果结构

后续扩平台时，默认要同时考虑两类输出：

1. 主结果表
- 固定 6 列：
  - `产品名称`
  - `核销人次`
  - `销售额`
  - `结算实付`
  - `采购金额`
  - `利润`

2. 双向订单差异检查结果
- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

当前默认差异结果要求：

- 两个方向都基于“第三方先按对账月份过滤后”的订单集合比较
- 两个方向都按唯一订单号输出，不重复
- `聚天下有、第三方当月无` 只展示 `订单号`
- `第三方有、聚天下当月无` 只展示 `第三方单号`

### 5.4 当前系统的真实边界

这一点必须明确：

当前业务设计真正适合的是“同一套主结果结构 + 同一套差异结果结构 + 同一套计算公式”的平台扩展。

也就是说，当前代码天然适合下面这种情况：

- 平台 Excel 不一样
- 但最终仍可统一成同样的对账输入
- 最终仍输出同样的主结果 6 列
- 最终仍输出同样的双向订单差异结果
- 最终仍使用同样的利润计算逻辑

当前代码并不天然支持下面这种情况：

- 平台输出列不同
- 平台差异结果展示结构不同
- 平台计算方式不同
- 平台需要专属结果表

如果未来遇到这类平台，不能只在适配器里打补丁，而要先升级“结果模型 + 汇总策略 + 导出逻辑”。

## 6. 标准新增流程

后续 Codex 新增平台时，默认按下面顺序执行。

### 6.1 先判断本次扩展属于哪一类

新增平台时，先归类，不要直接动代码。

#### A 类：仅文件结构不同

特征：

- Excel 工作表不同
- 字段名不同
- 日期列不同
- 一单多行的聚合方式不同
- 但最终主结果表、差异结果结构和计算公式都与当前一致

处理方式：

- 只新增平台适配器
- 注册平台
- 开放前端选项
- 补测试

#### B 类：输出列相同，但计算公式不同

特征：

- 仍然输出类似 6 列结果
- 仍然输出同样的双向订单差异结果
- 但公式不再完全复用当前领域算法

例如：

- `销售额` 和 `结算实付` 来自不同字段
- `利润` 不是简单相减
- 某些指标来自外部平台额外列

处理方式：

- 新增平台适配器
- 同时新增“平台专属汇总策略”或“平台专属计算器”
- 不要把复杂平台公式硬编码进当前通用算法里

#### C 类：输出列和结果结构都不同

特征：

- 结果表不是固定 6 列
- 或差异结果结构不再是当前默认的两个区域
- 需要额外列、少列或多张表
- Excel 导出结构也不同

处理方式：

- 新增平台适配器
- 升级统一结果模型
- 升级结果页渲染方式
- 升级 Excel 导出器
- 再补平台专属测试

### 6.2 推荐的结构升级方向

如果未来进入 B 类或 C 类扩展，不建议继续沿用“一个固定 `ProductSummaryRow` 打天下”的方式。

推荐把系统拆成两层扩展点：

1. `平台适配器`
负责读取平台原始 Excel，输出统一或半统一的基础数据。

2. `对账规则/报表策略`
负责定义该平台：
- 如何汇总
- 输出哪些列
- 差异结果怎么展示
- 每列怎么计算
- Excel 导出如何排版

换句话说：

- 适配器解决“怎么读平台文件”
- 策略层解决“怎么算、怎么展示”

如果只把所有差异都塞进适配器，后面平台一多，代码会很难维护。

### 6.3 第一步：研究样例文件

先看真实 Excel，不要先写代码。

至少确认以下信息：

- 工作表名称
- 订单关联字段
- 金额字段
- 月份过滤字段
- 差异检查使用哪个订单字段做集合比较
- 是否存在一单多行
- 是否有退款/负数
- 订单号是否会被 Excel 读成 `12345.0`
- 是否存在空行、合并单元格、前置说明行

建议先写成简短分析结论，再开始编码。

### 6.4 第二步：判断是否可以复用当前领域模型

如果新平台满足以下条件，可以直接复用当前领域层：

- 能映射出 `external_order_no`
- 能得到一个统一的金额字段
- 能按月份过滤
- 最终仍然按产品维度输出同样 6 列
- 最终仍然输出当前默认的双向订单差异结果

如果只是 A 类扩展，可以直接沿用当前领域模型。

如果已经进入 B 类或 C 类扩展，就不要直接复用当前 `ProductSummaryRow + reconcile_orders()` 组合，而应该先设计新的策略层或结果模型。

## 7. 推荐的适配器编写模板

新增平台时，建议从这个骨架出发：

```python
import pandas as pd

from app.infrastructure.date_parser import month_date_range
from app.models.reconciliation import ExternalOrderAggregate, PlatformParseResult
from app.platforms.base import PlatformAdapter


class NewPlatformAdapter(PlatformAdapter):
    platform_name = "new_platform"
    worksheet_name = "待确认工作表名"

    def parse(self, dataframe: pd.DataFrame, reconciliation_month: str) -> PlatformParseResult:
        order_column = "待确认订单字段"
        amount_column = "待确认金额字段"
        date_column = "待确认日期字段"
        required_columns = [order_column, amount_column, date_column]

        missing_columns = [column for column in required_columns if column not in dataframe.columns]
        if missing_columns:
            raise ValueError(f"文件缺少必要字段: {', '.join(missing_columns)}")

        working = dataframe[dataframe[order_column].notna()].copy()
        working[order_column] = working[order_column].astype(str).str.replace(".0", "", regex=False)
        working[amount_column] = pd.to_numeric(working[amount_column], errors="coerce").fillna(0)
        working["_business_date"] = pd.to_datetime(working[date_column], errors="coerce")

        start_date, next_month_start = month_date_range(reconciliation_month)
        in_month_mask = (
            working["_business_date"].notna()
            & (working["_business_date"] >= pd.Timestamp(start_date))
            & (working["_business_date"] < pd.Timestamp(next_month_start))
        )
        filtered_out_count = int((~in_month_mask).sum())
        filtered = working.loc[in_month_mask].copy()

        grouped = (
            filtered.groupby(order_column, as_index=False)
            .agg(
                {
                    amount_column: "sum",
                    "_business_date": "min",
                }
            )
            .rename(columns={order_column: "external_order_no"})
        )
        row_counts = filtered.groupby(order_column).size().to_dict()

        orders = []
        for _, row in grouped.iterrows():
            order_no = str(row["external_order_no"])
            business_date = row["_business_date"].date() if pd.notna(row["_business_date"]) else None
            orders.append(
                ExternalOrderAggregate(
                    external_order_no=order_no,
                    settlement_amount=float(row[amount_column]),
                    platform_name=self.platform_name,
                    source_row_count=int(row_counts[order_no]),
                    business_date=business_date,
                )
            )

        return PlatformParseResult(
            orders=orders,
            filtered_out_of_month_row_count=filtered_out_count,
        )
```

说明：

- 这是模板，不是必须逐字复制
- 如果该平台有多种日期格式或复杂前置清洗，可以在适配器内部增加私有辅助方法
- 但不要把平台私有逻辑泄漏到领域层

## 8. 什么时候不该改领域层或结果模型

以下情况通常不需要改：

- 只是字段名不同
- 只是工作表名不同
- 只是日期列不同
- 只是外部平台一单多行，需要先聚合
- 只是新增一个平台选项

这类变化应该在适配器里解决，不应该去改：

- `reconcile.py`
- `ProductSummaryRow`
- 订单差异结果默认结构
- 页面固定列结构
- Excel 导出列定义

## 9. 什么时候可以改领域层

只有当统一业务模型发生变化时，才考虑改：

1. 输出列定义变化
例如不再是固定 6 列，或者要增加新指标。

2. 金额模型变化
例如新平台必须区分：
- 销售额
- 结算实付

且二者不能再共用 `settlement_amount`。

3. 匹配模型变化
例如不再是“订单号对订单号”，而是要按子订单号、券码、核销码等匹配。

4. 汇总粒度变化
例如不再按 `产品内容` 汇总，而要按 `产品 + 日期` 或 `产品 + 店铺` 汇总。

5. 导出结构变化
例如需要：
- 多个结果表
- 多个工作表
- 平台专属列顺序
- 页面动态列展示

6. 差异检查结构变化
例如需要：
- 不再按订单号集合比较
- 差异结果展示更多字段
- 差异结果需要更多方向
- 差异结果需要平台专属汇总

遇到这些情况时，正确顺序是：

1. 先调整数据模型
2. 再调整汇总策略或领域算法
3. 再调整适配器输出
4. 再调整页面和导出
5. 最后补齐测试

不要跳步骤。

## 10. 新平台接入的最小改动清单

这个清单只适用于 A 类扩展，也就是“文件格式不同，但输出结构和公式仍然相同”。

普通新增平台，通常只需要改这些地方：

- 新增 `app/platforms/<platform>_adapter.py`
- 修改 `app/platforms/registry.py`
- 修改 `app/web/routes.py`
- 修改 `app/web/templates/index.html`
- 新增 `tests/platforms/test_<platform>_adapter.py`
- 新增 `tests/integration/test_<platform>_reconciliation.py`
- 视情况修改 `tests/web/test_reconcile_route.py`

如果改动超出这个范围很多，先回头检查：

- 是不是已经进入 B 类或 C 类扩展
- 是不是把平台公式写进了错误层级
- 是不是该先抽“平台专属汇总策略”

## 11. 推荐测试策略

### 11.1 适配器单测必须覆盖

- 订单号能正确清洗
- 金额列能转成数值

- 断言匹配订单数
- 断言汇总产品数
- 断言至少 1 个关键产品的金额结果
- 断言 `聚天下有、第三方当月无` 的数量或关键订单号
- 断言 `第三方有、聚天下当月无` 的数量或关键订单号

- 平台专属公式测试

## 13. Codex 实操顺序建议

后续 Codex 如果要新增平台，建议严格按下面顺序：

1. 阅读本文档
2. 阅读新平台样例 Excel
3. 阅读当前 `ctrip_adapter.py`
4. 先判断属于 A 类、B 类还是 C 类扩展
5. 判断是否需要升级统一数据模型、汇总策略、差异结果结构和导出结构
6. 先写测试，再写代码
7. 先完成平台适配，再处理汇总策略、差异结果和前端展示
8. 跑完整测试
9. 再汇报结果

## 14. 一句话原则

新增平台时，先判断“这是文件接入差异，还是规则差异”；只有前者才优先“新增适配器”，后者要先升级主结果模型、差异结果结构和汇总策略，再动代码。
