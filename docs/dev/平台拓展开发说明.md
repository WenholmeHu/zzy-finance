# 平台拓展开发说明

## 1. 文档目的

本文档面向后续继续接平台的人，包括：

- AI 开发
- 人工开发
- 维护者

它的目标不是重复业务口径，而是回答下面几个更关键的问题：

1. 新平台应该接到哪里
2. 改某个平台逻辑时，优先改哪一层
3. 什么情况下只改适配器就够
4. 什么情况下必须升级领域层、页面或导出层
5. 新平台接入的推荐顺序和测试策略是什么

业务口径仍以专项规则文档为准；本文重点描述“如何在当前代码中落地”。

## 2. 先理解当前系统，而不是套旧模板

当前系统已经支持：

- `聚天下 + 携程`
- `聚天下 + 美团`
- `聚天下 + 抖音`

当前真实架构不是“一个适配器包打天下”，而是下面 5 层：

1. `PlatformSpec`
2. `PlatformAdapter`
3. `PlatformReportDefinition`
4. `Application Service`
5. `Domain Reconcile`

如果不先判断需求落在哪一层，后续很容易把平台私有逻辑写错位置。

## 3. 当前扩展点总览

### 3.1 PlatformSpec：平台静态元数据

代码位置：

- [app/platforms/base.py](D:/DaMoXing/ZZY/finance/app/platforms/base.py)
- [app/platforms/registry.py](D:/DaMoXing/ZZY/finance/app/platforms/registry.py)

- `internal_difference_label`

- 平台要读取哪些工作表

- 差异区显示什么表头
- 平台下拉框显示什么中文名

### 3.2 PlatformAdapter：平台原始文件解析

代码位置：

- [app/platforms/ctrip_adapter.py](D:/DaMoXing/ZZY/finance/app/platforms/ctrip_adapter.py)
- [app/platforms/meituan_adapter.py](D:/DaMoXing/ZZY/finance/app/platforms/meituan_adapter.py)
- [app/platforms/douyin_adapter.py](D:/DaMoXing/ZZY/finance/app/platforms/douyin_adapter.py)

统一接口：

```python
class PlatformAdapter(ABC):
    platform_name: str

    @abstractmethod
    def parse_workbook(
        self,
        workbook_data: dict[str, pd.DataFrame],
        reconciliation_month: str,
    ) -> PlatformParseResult:
        ...
```

适配器负责：

1. 字段校验
2. 日期解析
3. 金额清洗
4. 按月过滤
5. 一单多行聚合
6. 输出统一订单级结构

适配器不负责：

- 页面展示
- 导出排版
- 通用利润公式
- 通用差异结果算法

### 3.3 PlatformReportDefinition：主结果表定义

代码位置：

- [app/platforms/report_definitions.py](D:/DaMoXing/ZZY/finance/app/platforms/report_definitions.py)

这里负责：

- 主结果表有哪些列
- 列顺序
- 列标题
- 数值列标记

如果某个平台只是“主结果表列不同”，优先改这里。

### 3.4 Application Service：共享流程编排

代码位置：

- [app/application/reconciliation_service.py](D:/DaMoXing/ZZY/finance/app/application/reconciliation_service.py)

这里负责：

1. 根据 `PlatformSpec` 选择聚天下关联字段
2. 加载聚天下 `订单列表`
3. 加载目标平台的工作簿数据
4. 调用适配器
5. 把统一结果交给领域层

如果需求涉及：

- 聚天下读取规则
- 工作簿加载方式
- 订单号保真
- 平台共享流程

优先看这里。

### 3.5 Domain：统一对账算法

代码位置：

- [app/domain/reconcile.py](D:/DaMoXing/ZZY/finance/app/domain/reconcile.py)

当前默认规则：

- `internal.order_no == external.external_order_no`
- 按 `product_name` 汇总
- 汇总内部：
  - `actual_people`
  - `purchase_amount`
- 汇总外部全部 `metrics`
- 统一计算：
  - `profit = settlement_paid - purchase_amount`
- 生成双向差异：
  - `internal_only_order_nos`
  - `external_only_order_nos`

如果新平台的需求会动到这些规则，不要再把问题当成普通适配器开发。

## 4. 当前 3 个平台的差异一览

| 平台 | 工作表 | 聚天下匹配字段 | 月份过滤字段 | 关键指标 |
| --- | --- | --- | --- | --- |
| 携程 | `流水` | `订单号` | `出发时间` | `sales_amount`、`settlement_paid` |
| 美团 | `订单详情` | `订单号` | `销售时间` | `sales_amount`、`technical_service_fee`、`merchant_coupon`、`settlement_paid` |
| 抖音 | `分账明细-正向-团购`、`分账明细-退款-团购` | `渠道订单号` | `核销时间` | `sales_amount`、`technical_service_fee`、`commission`、`service_provider_commission`、`settlement_paid` |

额外注意：

- 携程、美团都走 `聚天下[订单号]`
- 抖音走 `聚天下[渠道订单号]`
- 抖音不是单工作表输入，而是双工作表合并
- 差异表头不是统一固定的，而是随平台变化

## 5. 新平台接入前必须先判断的 8 件事

### 5.1 聚天下用哪个字段匹配

不要默认都用 `订单号`。

当前已经存在两种模式：

- 默认模式：`订单号`
- 抖音模式：`渠道订单号`

如果新平台匹配字段不同，先在 `PlatformSpec` 明确。

### 5.2 平台需要读取几个工作表

当前系统支持：

- 单工作表
- 多工作表

如果新平台需要多个工作表，不要再退回旧的单 `worksheet_name` 思路。

### 5.3 用哪个日期字段做月份过滤

新增平台前必须明确：

- 日期字段名
- 日期格式
- 空值和异常值处理
- 过滤后差异集合怎么比较

### 5.4 订单级指标能否表达为 `metrics: dict[str, float]`

如果可以，当前模型通常够用。

如果不可以，例如：

- 需要多个阶段结果
- 需要非求和型指标
- 需要订单外再细分子粒度

那么要先升级模型。

### 5.5 主结果表是否仍然是单表

当前系统支持“单主结果表 + 动态列定义”。

如果未来出现：

- 多个主结果表
- 多个结果区
- 多工作表主结果导出

那就不是简单补 `report_definitions.py` 了。

### 5.6 差异结果是否仍然是双向集合比较

当前差异结构固定为：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

如果未来要求：

- 展示更多字段
- 超过两个方向
- 不是按订单号集合比较

则要升级结果模型与导出结构。

### 5.7 汇总粒度是否仍然按 `产品内容`

当前领域层默认：

- `InternalOrder.product_name`
- 最终按 `product_name` 汇总

如果新平台要求按：

- 产品 + 日期
- 产品 + 店铺
- 产品 + 渠道

则要升级领域层。

### 5.8 利润公式是否仍然通用

当前利润统一是：

```text
利润 = 结算实付 - 采购金额
```

如果新平台利润口径不同，不能把它偷偷塞进单个平台适配器里。

## 6. 改动路由表：遇到什么需求先改哪里

| 需求 | 优先修改位置 | 一般不该先动的位置 |
| --- | --- | --- |
| 改平台工作表名 | `registry.py` | `reconcile.py` |
| 改聚天下匹配字段 | `registry.py` | 各平台适配器 |
| 改差异表头 | `registry.py` | `reconcile.py` |
| 改平台字段映射 | 对应 `*_adapter.py` | `routes.py` |
| 改平台月份过滤字段 | 对应 `*_adapter.py` | `reconcile.py` |
| 改平台主结果列 | `report_definitions.py` | 领域层 |
| 改共享订单读取逻辑 | `reconciliation_service.py` | 单个平台适配器 |
| 改导出排版 | `excel_writer.py` | 适配器 |
| 改利润公式/汇总粒度 | `reconcile.py` | 仅 `report_definitions.py` |

## 7. 新平台接入推荐流程

### 7.1 先看真实样例，不要先写代码

至少确认：

- 工作表名
- 关联字段
- 日期字段
- 金额字段
- 是否存在退款 / 冲正 / 负数
- 是否一单多行
- 差异比较用哪个订单字段
- 订单号是否会出现 `12345.0`

### 7.2 先分类：A / B / C

#### A 类：输入结构不同，但结果结构兼容

通常只需要：

- 新增适配器
- 注册平台
- 补报表定义
- 补测试

#### B 类：指标或计算更复杂，但仍是单主结果表

通常需要：

- 新增适配器
- 扩充 `metrics`
- 视情况调整共享流程
- 补平台专属测试

#### C 类：结果结构变化

例如：

- 多主结果表
- 多导出工作表
- 差异结构改变

这类需求必须先升级：

- 结果模型
- 页面结构
- 导出结构
- 领域算法

### 7.3 再写代码

推荐顺序：

1. 适配器测试
2. 适配器实现
3. 注册平台规格
4. 报表定义
5. Web / 导出联动
6. 集成测试
7. 文档更新

## 8. 当前推荐的新增平台改动清单

新增一个普通平台时，通常至少会动到：

- `app/platforms/<platform>_adapter.py`
- `app/platforms/registry.py`
- `app/platforms/report_definitions.py`
- `tests/platforms/test_<platform>_adapter.py`
- `tests/integration/test_<platform>_reconciliation.py`

如果平台列结构或导出结构有变化，通常还会动到：

- `tests/web/test_reconcile_route.py`
- `tests/web/test_export_route.py`

## 9. 推荐适配器骨架

```python
import pandas as pd

from app.infrastructure.date_parser import month_date_range
from app.models.reconciliation import ExternalOrderAggregate, PlatformParseResult
from app.platforms.base import PlatformAdapter


class NewPlatformAdapter(PlatformAdapter):
    platform_name = "new_platform"

    def parse_workbook(
        self,
        workbook_data: dict[str, pd.DataFrame],
        reconciliation_month: str,
    ) -> PlatformParseResult:
        dataframe = workbook_data["待确认工作表"]

        order_column = "待确认订单字段"
        date_column = "待确认日期字段"
        required_columns = [order_column, date_column]
        missing_columns = [column for column in required_columns if column not in dataframe.columns]
        if missing_columns:
            raise ValueError(f"文件缺少必要字段: {', '.join(missing_columns)}")

        working = dataframe[dataframe[order_column].notna()].copy()
        working[order_column] = (
            working[order_column].astype(str).str.replace(".0", "", regex=False)
        )
        working["_business_date"] = pd.to_datetime(working[date_column], errors="coerce")

        metric_columns = {
            "sales_amount": "待确认销售额字段",
            "settlement_paid": "待确认结算字段",
        }
        for source_column in metric_columns.values():
            working[source_column] = pd.to_numeric(working[source_column], errors="coerce").fillna(0)

        start_date, next_month_start = month_date_range(reconciliation_month)
        in_month_mask = (
            working["_business_date"].notna()
            & (working["_business_date"] >= pd.Timestamp(start_date))
            & (working["_business_date"] < pd.Timestamp(next_month_start))
        )
        filtered_out_count = int((~in_month_mask).sum())
        filtered = working.loc[in_month_mask].copy()

        grouped = (
            filtered.groupby(order_column, as_index=False)
            .agg({**{column: "sum" for column in metric_columns.values()}, "_business_date": "min"})
            .rename(columns={order_column: "external_order_no"})
        )
        row_counts = filtered.groupby(order_column).size().to_dict()

        orders = []
        for _, row in grouped.iterrows():
            order_no = str(row["external_order_no"])
            orders.append(
                ExternalOrderAggregate(
                    external_order_no=order_no,
                    metrics={
                        metric_key: float(row[source_column])
                        for metric_key, source_column in metric_columns.items()
                    },
                    platform_name=self.platform_name,
                    source_row_count=int(row_counts[order_no]),
                )
            )

        return PlatformParseResult(
            orders=orders,
            filtered_out_of_month_row_count=filtered_out_count,
        )
```

## 10. 测试建议

### 10.1 适配器测试

至少覆盖：

- 缺工作表
- 缺字段
- 非当月过滤
- 一单多行聚合
- 金额映射
- 订单号清洗

### 10.2 集成测试

至少覆盖：

- 用真实样例跑完整流程
- 匹配订单数
- 汇总产品数
- 至少一个关键产品的指标
- 差异数量或关键差异订单号

### 10.3 Web / 导出测试

至少覆盖：

- 平台能在页面下拉框出现
- 主结果表头正确
- 差异表头正确
- 导出列顺序正确

## 11. 什么时候不能只改适配器

出现下面任意一种情况，就应该先停下来升级设计：

- 匹配规则不再是“订单号 = 订单号”
- 汇总维度不再是 `产品内容`
- 利润公式不再统一
- 主结果表不止一个
- 差异结果不止两个方向
- 页面需要多个结果区
- 导出需要多个主结果工作表
