## 技术决策
| 决策 | 理由 |
|------|------|
| 后续开发流程按“项目骨架 -> 核心引擎 -> 页面集成 -> 验证交付”组织 | 适合当前项目规模，也利于逐步验证 |
| 先把平台扩展能力做成结构预留，不实现飞猪/美团逻辑 | 当前目标仍然是携程对账上线 |
| 将测试分为单元测试、服务集成验证、人工页面验收 | 兼顾 Codex 自动化与财务实际使用场景 |
| 第一轮正式开发只实现阶段 1 | 先拿到稳定、可运行、可测试的工程骨架 |
| 双向订单差异检查与主结果表一并返回 | 这是最新确认的正式业务输出，不再忽略未匹配订单 |
| 差异结果在页面展示，不进入 Excel 导出 | 符合最新财务使用方式，财务在网页核对即可 |
| 结果页采用“财务核对优先型”而不是纯美观型改版 | 更符合真实使用场景，先看异常数量，再看主表，再查差异明细 |
| 差异明细编号只出现在网页，不进入 Excel | 符合最新财务使用方式，导出保持主结果更简洁 |
