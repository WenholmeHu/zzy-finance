# 样例数据与测试策略

## 1. 当前 `test_data` 状态

当前仓库默认保留的真实样例文件为：

- `test_data/jutianxia_tc.xlsx`
- `test_data/tongcheng.xlsx`

这表示当前默认维护口径偏向同程场景。

## 2. 当前测试分层

### 2.1 领域层测试

- 位置：`tests/domain/test_reconcile.py`
- 目标：验证通用匹配、汇总、利润、差异规则

### 2.2 平台适配器测试

- 位置：`tests/platforms/`
- 目标：验证平台级字段清洗、月份过滤、订单聚合、指标映射

### 2.3 集成测试

- 位置：`tests/integration/`
- 目标：验证从 Excel 输入到最终对账结果的完整流程

当前分两类：

- 构造数据集成测试
  - 例如 `tests/integration/test_sample_reconciliation.py`
  - 不依赖 `test_data/`
- 真实样例集成测试
  - 例如抖音 / 美团 / 同程的真实文件测试
  - 依赖 `test_data/`

### 2.4 Web 测试

- 位置：`tests/web/`
- 目标：验证首页、上传对账、导出结果

## 3. 当前稳定验证命令

```powershell
pytest tests -q
```

这是当前仓库最推荐的统一回归命令。

## 4. 为什么会出现 `skipped`

部分真实样例集成测试会先检查 `test_data/` 中是否存在目标文件。

当前会自动 `skip` 的典型情况是：

- 缺少 `test_data/jutianxia.xlsx` 或 `test_data/douyin.xlsx`
- 缺少 `test_data/jutianxia.xlsx` 或 `test_data/meituan.xlsx`

这是预期行为，不代表代码一定有问题。

同程当前默认样例已随仓库提供，因此 `tests/integration/test_tongcheng_reconciliation.py` 正常情况下不会因为样例缺失而跳过。

## 5. 当前测试口径的实际含义

- 携程
  - 有适配器测试
  - 有构造数据集成测试
  - 当前没有基于 `test_data/` 的独立真实样例集成测试
- 美团
  - 有适配器测试
  - 有真实样例集成测试，但依赖样例文件是否存在
- 抖音
  - 有适配器测试
  - 有真实样例集成测试，但依赖样例文件是否存在
- 同程
  - 有适配器测试
  - 有真实样例集成测试
  - 当前默认样例已在仓库中

## 6. 新增平台时建议的测试补法

至少补 4 类：

1. 平台适配器测试
2. 集成测试
3. Web 上传/结果测试
4. 视导出结构变化补导出测试

推荐顺序：

1. 先写适配器测试
2. 再写集成测试
3. 最后补 Web 和导出测试

## 7. 管理 `test_data` 的建议

- 尽量保留命名清晰的真实样例文件
- 如果切换了默认样例，请同步更新 `README.md` 和交接文档
- 如果样例涉及真实业务数据，建议先做脱敏
- 不要只删样例不改测试说明，否则后续维护人员会误判 `skip`

## 8. 手工复核与自动测试的关系

自动测试可以覆盖：

- 匹配规则是否正确
- 平台字段映射是否正确
- 页面和导出是否基本可用

但自动测试不能完全替代人工验收，尤其是：

- 真实业务月份边界
- 特殊退款或负数行
- 财务最终认可的口径

建议配合 [06-人工验收清单.md](./06-人工验收清单.md) 一起使用。
