# 维护与排障

## 1. 排障的推荐顺序

遇到“结果不对”时，建议按下面顺序排：

1. 先确认业务规则和真实样例字段
2. 再确认平台适配器是否按正确月份字段过滤
3. 再确认聚天下匹配字段是否选对
4. 再确认共享领域层是否适合当前业务规则
5. 最后再看页面或导出是否只是展示问题

## 2. 常见问题速查

| 现象 | 常见原因 | 第一优先检查位置 |
| --- | --- | --- |
| 报“缺少工作表” | `registry.py` 的工作表配置与真实 Excel 不一致 | `app/platforms/registry.py` |
| 单工作表平台读取失败 | 工作表模式应为 `single_dynamic`，但配置错了 | `app/platforms/registry.py`、`app/application/reconciliation_service.py` |
| 某月结果为空或明显偏少 | 月份过滤字段选错，或日期解析异常 | 目标平台 `*_adapter.py` |
| 明明有订单却匹配不上 | 聚天下匹配字段不对，或订单号保真失败 | `registry.py`、`reconciliation_service.py` |
| 大订单号变成 `.0` 或科学计数法 | Excel 读入后字符串保真处理没覆盖 | `app/application/reconciliation_service.py` |
| 一单多行金额翻倍 | 平台适配器聚合逻辑不正确 | 目标平台 `*_adapter.py` |
| 主结果列顺序或列名不对 | 平台报表定义没同步 | `app/platforms/report_definitions.py` |
| 页面表头对，但导出 Excel 不对 | 导出层没有同步列定义 | `app/infrastructure/excel_writer.py`、`app/web/routes.py` |
| 差异订单明显异常 | 共享匹配与差异算法需要核查 | `app/domain/reconcile.py` |
| 某些测试显示 `skipped` | `test_data` 中缺少该平台真实样例文件 | `tests/integration/*`、`test_data/` |

## 3. 什么时候先查平台适配器

以下情况大多数先查 `app/platforms/*_adapter.py`：

- 平台字段名变了
- 月份过滤字段变了
- 某平台多了退款表或说明表
- 同一订单需要先合并再参与对账
- 某个平台金额字段映射变了

## 4. 什么时候不是平台 bug，而是架构边界到了

以下情况通常不是“改两行适配器”能解决的：

- 平台要多个主结果表
- 平台要按产品之外的维度汇总
- 利润公式平台专属
- 匹配规则不再是订单号对订单号
- 差异结果不再是双向订单集合

这类需求要先看：

- [项目设计文档.md](../dev/项目设计文档.md)
- [平台拓展开发说明.md](../dev/平台拓展开发说明.md)

## 5. 推荐排障命令

先跑完整基线：

```powershell
pytest tests -q
```

如果只怀疑某个平台，优先缩小到平台测试：

```powershell
pytest tests/platforms/test_tongcheng_adapter.py -q
pytest tests/integration/test_tongcheng_reconciliation.py -q
```

如果怀疑页面或导出：

```powershell
pytest tests/web/test_reconcile_route.py -q
pytest tests/web/test_export_route.py -q
```

## 6. 当前排障方式的现实情况

当前仓库没有专门的结构化日志体系，排障主要依赖：

- 真实样例复现
- 平台适配器测试
- 集成测试
- Web 测试

如果后续公司长期使用，建议补充更明确的错误日志与输入上下文记录。
