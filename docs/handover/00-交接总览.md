# 交接总览

## 1. 项目一句话

这是一个面向财务人员使用的本地网页对账工具，用来完成 `聚天下 + 外部平台` 的月度订单对账，并生成页面结果与 Excel 导出。

当前已支持：

- `聚天下 + 携程`
- `聚天下 + 美团`
- `聚天下 + 抖音`
- `聚天下 + 同程`

## 2. 当前系统能做什么

- 网页上传两份 Excel
- 按月过滤外部平台数据
- 以订单号规则完成平台与聚天下订单匹配
- 匹配成功后按产品汇总
- 展示主结果表
- 展示内部单边差异、外部单边差异
- 导出主结果与差异结果 Excel

## 3. 当前系统的真实边界

当前架构已经支持：

- 不同平台使用不同工作表
- 单工作表但工作表名动态变化
- 平台级聚天下匹配字段切换
- 平台级订单指标集合差异
- 平台级主结果表列结构差异
- 平台级差异表头差异

但当前仍然默认：

- 只有 `1` 个主结果表
- 只有 `2` 个方向的差异结果
- 匹配模型仍然是“内部订单号 = 外部订单号”
- 汇总维度仍然是 `产品内容`
- 利润公式仍然是 `结算实付 - 采购金额`

如果未来新平台超出这些边界，不应只补适配器，而要同步评估领域层、页面和导出结构。

## 5. 第一天接手建议

建议按下面顺序快速建立把控：

1. 跑一次完整测试：`pytest tests -q`
2. 启动本地服务：`uvicorn app.main:app --reload`
3. 用当前默认样例走一遍网页流程
4. 读目标平台对应的适配器与测试
5. 再开始改代码

## 6. 最关键的代码入口

| 位置 | 作用 |
| --- | --- |
| `app/platforms/registry.py` | 平台注册、工作表模式、聚天下匹配字段、差异表头 |
| `app/platforms/*_adapter.py` | 平台 Excel 清洗、按月过滤、订单聚合、指标映射 |
| `app/platforms/report_definitions.py` | 主结果表列定义 |
| `app/application/reconciliation_service.py` | 对账编排、Excel 读取、聚天下订单加载 |
| `app/domain/reconcile.py` | 通用对账算法、汇总、利润、差异结果 |
| `app/infrastructure/excel_writer.py` | 主结果和差异表导出 |
| `app/web/routes.py` | 页面上传、结果渲染、导出路由 |

## 7. 当前样例与测试口径

当前仓库默认的真实样例在 `test_data/`：

- `jutianxia_tc.xlsx`
- `tongcheng.xlsx`

这意味着：

- 当前最方便做手工复核的是同程
- 抖音 / 美团基于真实样例文件的集成测试，在样例缺失时会自动跳过
- 携程当前主要依赖构造数据测试，而不是 `test_data/` 中的真实样例文件

更多细节见 [03-样例数据与测试策略.md](./03-样例数据与测试策略.md)。

## 8. 后续扩平台时最重要的提醒

- 先拿真实样例，再决定改哪一层
- 先确认月份过滤字段，再确认订单匹配字段
- 不要上来就抄现有适配器，先判断目标平台属于 A 类、B 类还是结构升级
- 平台差异如果只是输入差异，优先改 `adapter + registry + report_definitions`
- 如果利润、汇总粒度、差异结构变了，要先评估共享领域层
