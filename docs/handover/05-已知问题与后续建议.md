# 已知问题与后续建议

## 1. 当前已知边界

当前系统虽然已经支持四个平台，但仍然有明确的结构边界：

- 只有一个主结果表
- 只有两个方向的差异结果
- 汇总维度固定为 `产品内容`
- 匹配模型固定为订单号对订单号
- 利润公式固定为 `结算实付 - 采购金额`

如果后续新平台不满足这些假设，通常需要做结构升级。

## 2. 当前测试与样例上的现实限制

- `test_data/` 当前只保留了同程真实样例
- 抖音 / 美团真实样例集成测试在缺样例时会自动跳过
- 携程当前没有基于 `test_data/` 的独立真实样例集成测试

这不影响当前代码运行，但会影响某些平台的真实回归覆盖度。

## 3. 当前运维形态的限制

- 项目是本地网页工具，不是集中部署服务
- 当前没有 Docker、安装包或一键启动脚本
- 当前没有专门的结构化日志体系

因此后续排障更依赖：

- 真实样例
- 自动化测试
- 人工复核

## 4. 文档上的注意事项

- `docs/dev/` 是开发设计文档
- `docs/handover/` 是交接维护文档
- `docs/history/` 是历史资料，不建议作为当前设计依据

后续如果切换样例、修改测试口径、扩新平台，记得同步更新这三处文档入口。

## 5. 强烈建议后续补的内容

1. 统一的业务联系人与字段确认人清单
2. 各平台真实样例的长期存档方式
3. 一份固定格式的人工验收记录模板
4. 更清晰的错误日志或失败上下文输出
5. 一个更稳定的启动脚本或交付脚本

## 6. 如果项目继续长期演进，优先建议做什么

### 6.1 样例数据管理升级

建议考虑把 `test_data/` 改为按平台分目录维护，而不是频繁整体替换。这样能降低真实样例被覆盖后的测试缺口。

### 6.2 日志能力升级

建议在应用编排层和平台适配层增加更清晰的错误上下文，例如：

- 当前平台
- 当前月份
- 实际读取到的工作表
- 缺失字段名

### 6.3 复杂平台接入前先做结构评估

如果未来要接的平台涉及：

- 多主结果表
- 平台专属利润公式
- 更复杂的差异类型

建议先出一份设计说明，再动代码。

## 7. 当前最容易被误解的一点

“已经支持多平台”不等于“后续所有平台都只要抄一个适配器”。

当前支持的是“在共享对账框架下接入若干结构兼容的平台”。一旦业务口径突破现有边界，核心共享层也需要一起演进。
