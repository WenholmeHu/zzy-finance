# 抖音对账扩展设计

文档日期：2026-04-15  
当前状态：设计已确认，待进入实施计划  
适用范围：在保留携程、美团现有行为的前提下，新增抖音平台对账能力，并支持抖音平台的双工作表输入、专属匹配键和专属主结果列

## 1. 背景与目标

当前系统已经支持：

- 聚天下 + 携程
- 聚天下 + 美团
- 平台动态主结果表
- 双向订单差异检查
- 页面展示与 Excel 导出

抖音平台与现有平台相比，新增了三个关键差异：

1. 输入文件不是单工作表，而是两张明细表共同参与对账
- `分账明细-正向-团购`
- `分账明细-退款-团购`

2. 内部关联键不是当前默认的 `聚天下[订单号]`
- 抖音规则为：`聚天下[渠道订单号] == 抖音[订单编号]`

3. 主结果表结构和指标集合都与现有平台不同
- 需要输出：
  - 产品名称
  - 核销人次
  - 销售额
  - 技术服务费
  - 佣金
  - 服务商佣金
  - 结算实付
  - 采购金额
  - 利润

因此，抖音扩展不能被视为“只新增一个 adapter”的 A 类扩展，但目前仍可保持：

- 单一主结果表
- 统一双向差异检查结构
- 统一按产品汇总的大方向

本次设计结论是：将抖音视为偏 `B 类扩展` 处理，即升级平台契约和输入读取方式，但暂不重构页面整体结构与领域层总体模式。

## 2. 已确认输入与业务口径

### 2.1 文件与工作表

- 聚天下文件：`test_data/jutianxia.xlsx`
- 聚天下工作表：`订单列表`
- 抖音文件：`test_data/douyin.xlsx`
- 抖音工作表：
  - `分账明细-正向-团购`
  - `分账明细-退款-团购`

已核对样例文件，以上两个抖音工作表在样例中真实存在。

### 2.2 已确认的聚天下字段

- `产品内容`
- `实到人数`
- `采购金额`
- `渠道订单号`

已核对样例文件，`渠道订单号` 列真实存在，可作为抖音对账的内部匹配字段。

### 2.3 已确认的抖音关键字段

两张抖音明细表中已确认存在以下关键字段：

- `核销时间`
- `订单编号`
- `订单实收金额`
- `增量宝`
- `软件服务费`
- `平台撮合佣金`
- `达人佣金`
- `撮合经纪服务费`
- `保险费用`
- `服务商佣金`
- `商家应得`

### 2.4 月份过滤规则

- 前端选择 `reconciliation_month`
- 抖音适配器按两张明细表中的 `核销时间` 做月份过滤
- 非目标月份记录不参与主结果计算
- 差异比较也必须基于“按 `核销时间` 过滤后的抖音订单编号集合”

月份过滤实现口径继续沿用当前系统统一规则：

- 包含月初
- 不包含下月月初
- 即 `[month_start, next_month_start)`

### 2.5 关联规则

抖音平台使用以下规则关联：

- `聚天下[渠道订单号] == 抖音[订单编号]`

这意味着：

- 当前系统默认使用的 `聚天下[订单号]` 不能直接复用到抖音
- 应用层或平台契约必须支持“按平台选择内部匹配字段”

### 2.6 差异表规则

抖音继续沿用当前双向差异检查结构：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

但比较主键需要调整为抖音专属口径：

- 内部侧：`聚天下[渠道订单号]`
- 外部侧：按 `核销时间` 过滤后的抖音 `订单编号` 去重集合

### 2.7 金额处理口径

已确认抖音金额处理规则为：

- 正向表与退款表中的相关字段，统一按原值直接相加
- 不对退款表做额外取负、翻转或反向处理

这点尤其重要，因为样例中退款表部分字段已经是负数，部分字段为正数，后续实现必须严格遵循“直接相加”的业务口径，而不是基于字段符号再次推导。

### 2.8 主结果表口径

主结果表第一列继续显示：

- `产品名称`

虽然汇总来源仍为聚天下 `产品内容`，但页面和导出表头保持为 `产品名称`，与现有平台风格一致。

## 3. 输出字段定义

抖音主结果表输出以下列：

- 产品名称
- 核销人次
- 销售额
- 技术服务费
- 佣金
- 服务商佣金
- 结算实付
- 采购金额
- 利润

各指标规则如下：

- `产品名称`
  - 来源：聚天下 `产品内容`
  - 作用：作为汇总维度

- `核销人次`
  - 来源：聚天下 `实到人数`
  - 规则：只统计成功匹配且抖音 `核销时间` 属于目标月份的订单

- `销售额`
  - 来源：抖音两张表的 `订单实收金额`
  - 规则：正向原值 + 退款原值

- `技术服务费`
  - 来源：抖音两张表的 `增量宝` 与 `软件服务费`
  - 规则：`正向.增量宝 + 正向.软件服务费 + 退款.增量宝 + 退款.软件服务费`

- `佣金`
  - 来源：抖音两张表的
    - `平台撮合佣金`
    - `达人佣金`
    - `撮合经纪服务费`
    - `保险费用`
  - 规则：上述字段在正向表和退款表中的原值直接相加

- `服务商佣金`
  - 来源：抖音两张表的 `服务商佣金`
  - 规则：正向原值 + 退款原值

- `结算实付`
  - 来源：抖音两张表的 `商家应得`
  - 规则：正向原值 + 退款原值

- `采购金额`
  - 来源：聚天下 `采购金额`
  - 规则：只统计成功匹配且抖音 `核销时间` 属于目标月份的订单

- `利润`
  - 公式：`结算实付 - 采购金额`

## 4. 设计结论

推荐采用“平台契约升级 + 抖音适配器 + 抖音报表定义”方案。

### 4.1 平台契约需要升级什么

当前平台接口默认假设：

- 一个平台只读取一张工作表
- 聚天下总是使用 `订单号` 作为内部匹配键

这两个假设都不适用于抖音。

建议平台元数据至少新增以下信息：

- 该平台需要读取哪些工作表
- 该平台使用聚天下哪个字段参与匹配
- 该平台差异结果的内部侧/外部侧表头文案

### 4.2 为什么不把抖音特判写进应用层

如果直接在 `reconciliation_service.py` 中写：

- 抖音就读两张表
- 抖音就改用 `渠道订单号`

虽然短期可跑通，但会把平台私有规则泄漏到主流程中。后续再接复杂平台时，应用层会快速退化成大量平台分支。

因此推荐做法是：

- 应用层只读取平台声明的工作表
- 应用层只根据平台声明选择内部匹配字段
- 具体“怎么合并两张抖音表、怎么算指标”全部留在抖音适配器中

## 5. 数据模型与流程调整

### 5.1 内部订单匹配键

领域层当前使用 `InternalOrder.order_no` 与外部订单做匹配。

为了避免改动领域层总体模式，推荐做法是：

- 应用层在读取聚天下时，根据当前平台选择匹配列
- 对于携程、美团：仍读取 `订单号`
- 对于抖音：读取 `渠道订单号`
- 读取后统一映射到 `InternalOrder.order_no`

这样领域层无需知道“本次到底是订单号还是渠道订单号”，只处理统一后的匹配键。

### 5.2 外部订单模型

当前系统已经使用：

- `external_order_no`
- `metrics`
- `platform_name`
- `source_row_count`
- `business_date`

抖音适配器可以直接复用这一模型，无需重新设计结果容器。

建议抖音订单级 `metrics` 输出如下：

- `sales_amount`
- `technical_service_fee`
- `commission`
- `service_provider_commission`
- `settlement_paid`

### 5.3 平台报表定义

抖音新增一套主结果表列定义：

- `product_name`
- `actual_people`
- `sales_amount`
- `technical_service_fee`
- `commission`
- `service_provider_commission`
- `settlement_paid`
- `purchase_amount`
- `profit`

页面和 Excel 导出继续复用当前动态列机制。

## 6. 抖音平台适配设计

### 6.1 读取方式

抖音适配器需要同时消费：

- `分账明细-正向-团购`
- `分账明细-退款-团购`

不建议只保留 `worksheet_name: str` 单字段接口，而应支持多工作表输入。

### 6.2 单表清洗规则

对两张表分别执行：

- 校验必要字段
- 过滤 `订单编号` 为空的行
- 清洗 `12345.0` 形式的订单号
- 将关键金额列使用 `pd.to_numeric(..., errors="coerce").fillna(0)` 做数值化
- 将 `核销时间` 使用 `pd.to_datetime(..., errors="coerce")` 解析
- 按 `[month_start, next_month_start)` 过滤目标月份

### 6.3 单表指标映射

对任一抖音明细行，统一映射为：

- `sales_amount = 订单实收金额`
- `technical_service_fee = 增量宝 + 软件服务费`
- `commission = 平台撮合佣金 + 达人佣金 + 撮合经纪服务费 + 保险费用`
- `service_provider_commission = 服务商佣金`
- `settlement_paid = 商家应得`

### 6.4 合并与聚合规则

完成单表映射后：

1. 将正向表和退款表的标准化明细行合并
2. 按 `订单编号` 聚合
3. 对所有指标求和
4. 使用最早 `核销时间` 作为 `business_date`
5. 使用原始参与聚合行数作为 `source_row_count`

这样可以兼容：

- 同一订单只出现在正向表
- 同一订单只出现在退款表
- 同一订单同时出现在两张表
- 同一订单在单张表内部有多行

## 7. Web 与导出调整

### 7.1 主结果表

主结果表继续复用当前动态列渲染机制，只需要在平台报表定义中新增抖音列即可。

### 7.2 差异结果标题与表头

当前差异结果结构可以继续复用，但表头建议支持平台专属文案：

- 抖音内部侧建议显示 `渠道订单号`
- 抖音外部侧建议显示 `订单编号`

这样页面和导出都能更准确地反映抖音的实际比对主键。

### 7.3 Excel 导出

主结果导出仍然沿用单工作表：

- `对账结果`

差异导出仍然沿用两个工作表：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

但差异工作表中的列标题应允许平台化。

## 8. 错误处理

抖音接入后需要至少覆盖以下错误场景：

- 缺少 `分账明细-正向-团购` 工作表
- 缺少 `分账明细-退款-团购` 工作表
- 任一工作表缺少必要字段
- `核销时间` 无法解析
- 聚天下文件缺少 `渠道订单号`

错误提示应延续当前项目风格，面向业务用户，例如：

- `抖音文件缺少工作表: 分账明细-退款-团购`
- `抖音文件[分账明细-正向-团购]缺少必要字段: 订单编号`
- `抖音文件[分账明细-退款-团购]中的“核销时间”存在无法识别的日期，请检查原始数据格式`
- `聚天下文件缺少“渠道订单号”列，无法进行抖音对账`

如果两张抖音表都成功读取，但目标月份过滤后无有效记录，则不应报系统错误，而应返回：

- 空主结果表
- 正常差异结果
- 正常摘要统计

## 11. 本次设计结论

本次抖音扩展建议采用“升级平台契约 + 新增抖音适配器 + 新增抖音报表定义”的方案。

原因如下：

- 能承接抖音双工作表输入
- 能支持抖音使用 `渠道订单号` 做内部匹配
- 统一 `metrics` 模型仍然可复用
- 主结果表仍然可由当前动态列机制表达
- 双向差异结构仍然可复用，只需平台化表头
- 不会把平台私有规则泄漏到应用层或领域层

设计确认后，下一步应进入实施计划编写，并按测试优先顺序推进，不直接跳入编码。
