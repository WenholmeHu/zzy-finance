# 同程对账扩展设计

文档日期：2026-04-15  
当前状态：设计已确认，待进入实施计划  
适用范围：在保留携程、美团、抖音现有行为的前提下，新增同程平台对账能力，并补足“单工作表但表名动态变化”的平台输入支持

## 1. 背景与目标

当前系统已经支持：

- 聚天下 + 携程
- 聚天下 + 美团
- 聚天下 + 抖音
- 平台动态主结果表
- 双向订单差异检查
- 页面展示与 Excel 导出

同程平台与现有平台相比，业务口径本身较简单，但在输入文件上有一个与现有平台不同的点：

1. 同程文件只有一张工作表
2. 这张工作表的名字不是固定常量，而是带日期变化
3. 业务计算仍然属于当前系统能直接表达的范围

因此，本次同程接入应被视为 `A 类扩展 + 一个很小的共享读取能力补强`：

- 不需要升级领域层
- 不需要调整页面结构
- 不需要调整导出模型
- 只需要补一个通用的“动态单工作表读取”能力

本次设计目标是：

- 新增 `同程` 平台接入
- 保持现有单主结果表、双向差异、订单号匹配和利润公式不变
- 让平台文件可以在“只有一张表，但名称动态”时仍被安全读取

## 2. 已确认输入与业务口径

### 2.1 文件与工作表

- 聚天下文件：`test_data/jutianxia_tc.xlsx`
- 聚天下工作表：`订单列表`
- 同程文件：`test_data/tongcheng.xlsx`
- 同程工作表：实际样例中只有一张表，名称为 `订单2026-04-01`

这里已经和需求再次确认：

- 聚天下侧正式使用 `订单列表`
- 不是 `订单详情`

### 2.2 已确认的聚天下字段

- `订单号`
- `产品内容`
- `实到人数`
- `采购金额`

### 2.3 已确认的同程关键字段

样例文件中已确认存在以下关键字段：

- `旅游日期`
- `应结(元)`
- `三方流水号`

此外还存在：

- `订单流水号`
- `订单确认号`
- `取票人`
- `门票类型`
- `结算单价(元)`
- `票数`

但这些字段不参与当前主结果计算。

### 2.4 月份过滤规则

- 前端选择 `reconciliation_month`
- 同程适配器按 `旅游日期` 做月份过滤
- 非目标月份记录不参与主结果计算
- 差异比较也必须建立在“按 `旅游日期` 过滤后的同程订单集合”之上

月份过滤口径继续沿用当前系统统一规则：

- 包含月初
- 不包含下月月初
- 即 `[month_start, next_month_start)`

### 2.5 关联规则

同程平台使用以下规则关联：

- `聚天下[订单号] == 同程[三方流水号]`

这意味着：

- 同程继续复用当前默认的聚天下匹配字段 `订单号`
- 不需要像抖音那样改为 `渠道订单号`

### 2.6 主结果表规则

同程主结果表继续沿用携程同类结构：

- 产品名称
- 核销人次
- 销售额
- 结算实付
- 采购金额
- 利润

### 2.7 差异表规则

同程继续沿用当前双向差异检查结构：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

表头文案应为：

- 内部侧：`订单号`
- 外部侧：`三方流水号`

## 3. 输出字段定义

同程主结果表输出以下列：

- 产品名称
- 核销人次
- 销售额
- 结算实付
- 采购金额
- 利润

各指标规则如下：

- `产品名称`
  - 来源：聚天下 `产品内容`
  - 作用：作为汇总维度

- `核销人次`
  - 来源：聚天下 `实到人数`
  - 规则：只统计成功匹配且同程 `旅游日期` 属于目标月份的订单

- `销售额`
  - 来源：同程 `应结(元)`
  - 规则：按订单匹配后，再按产品汇总求和

- `结算实付`
  - 来源：同程 `应结(元)`
  - 规则：与 `销售额` 完全一致，可直接复用相同值

- `采购金额`
  - 来源：聚天下 `采购金额`
  - 规则：只统计成功匹配且同程 `旅游日期` 属于目标月份的订单

- `利润`
  - 公式：`结算实付 - 采购金额`

## 4. 样例数据核对结论

基于当前仓库中的真实样例文件，已完成一次开发前核对，得到以下结论：

- `tongcheng.xlsx` 实际只有 1 张工作表
- 该工作表名动态变化，不适合在注册表中写死固定名称
- 样例平台记录总数：`738`
- 2026-03 范围内有效记录数：`737`
- 因非目标月份或无有效日期被过滤记录数：`1`
- 2026-03 范围内的 `三方流水号` 唯一订单数：`737`
- 当前样例中同程侧无外部差异订单：`external_only_count = 0`
- 按现有领域规则汇总后，主结果表产品数为：`23`

样例中可作为真实链路断言的代表性产品包括：

- `【即买即用】【线下扫码】东运旅行卢宅景区成人票+卢宅文创产品1份`
  - 核销人次：`257`
  - 销售额：`16705.0`
  - 结算实付：`16705.0`
  - 采购金额：`13364.0`
  - 利润：`3341.0`

## 5. 设计结论

推荐采用“共享单表读取能力补强 + 同程适配器 + 同程报表定义”方案。

### 5.1 为什么同程不需要改领域层

同程当前仍然满足系统的既有边界：

- 匹配规则仍然是订单号对订单号
- 汇总维度仍然是 `产品内容`
- 利润公式仍然统一
- 主结果仍然只有一个
- 差异结果仍然只有两个方向

因此，`app/domain/reconcile.py` 和 `app/models/reconciliation.py` 不应作为这次扩展的优先改动点。

### 5.2 为什么需要补共享读取层

当前平台注册信息默认用固定 `worksheet_names` 描述需要读取的工作表。

这适用于：

- 携程：固定单表
- 美团：固定单表
- 抖音：固定双表

但不适用于同程这种“只有一张表，但工作表名动态变化”的平台。

如果把同程写成应用层特判，例如：

- `if platform == "tongcheng": 读取第一张表`

虽然短期可跑通，但平台私有规则会泄漏到共享流程中，不利于后续复用。

因此建议在共享层补一个小能力：

- 支持某个平台声明“平台文件必须只有一张工作表，且工作表名不固定”
- 应用层统一处理这个声明
- 同程适配器只处理业务字段，不负责 Excel 工作表探测

### 5.3 推荐的元数据表达方式

平台元数据建议补一个“工作表选择模式”，支持至少两种情况：

- 固定工作表列表
- 动态单工作表

同程可声明为：

- 平台文件应当只有一张工作表
- 应用层读取该唯一工作表，并以真实工作表名作为键传给适配器

如果文件里存在多张工作表，则应抛出清晰错误，而不是默默取第一张。

## 6. 同程平台适配设计

### 6.1 读取方式

同程适配器接收来自共享层的 `workbook_data: dict[str, pd.DataFrame]`。

适配器不依赖固定工作表名，只要求：

- `workbook_data` 中恰好有一张同程明细表

随后从这张表中读取业务字段并完成统一结构转换。

### 6.2 字段校验

至少校验以下字段存在：

- `旅游日期`
- `应结(元)`
- `三方流水号`

缺字段时，应报出清晰错误，便于业务人员判断是模板不对还是导出内容不完整。

### 6.3 清洗与过滤规则

对同程明细执行：

- 过滤 `三方流水号` 为空的行
- 清洗 `12345.0` 形式的订单号
- `旅游日期` 使用 `pd.to_datetime(..., errors="coerce")`
- `应结(元)` 使用 `pd.to_numeric(..., errors="coerce").fillna(0)`
- 按 `[month_start, next_month_start)` 过滤目标月份

### 6.4 指标映射

对任一过滤后的同程订单，输出统一指标：

```python
metrics = {
    "sales_amount": row["应结(元)"],
    "settlement_paid": row["应结(元)"],
}
```

若未来出现同一个 `三方流水号` 多行，则应先按订单聚合：

- `应结(元)` 求和
- `business_date` 取最早日期
- `source_row_count` 记录参与聚合的明细行数

## 7. 页面与导出设计

### 7.1 主结果表

同程主结果表继续复用当前动态列渲染机制，只需在平台报表定义中增加同程列定义。

推荐直接复用携程的列结构：

- `product_name`
- `actual_people`
- `sales_amount`
- `settlement_paid`
- `purchase_amount`
- `profit`

### 7.2 差异结果标题与表头

差异结果区域继续复用当前双向结构，但表头需要在平台注册信息中配置为：

- 内部侧：`订单号`
- 外部侧：`三方流水号`

页面与差异导出都复用这套表头。

### 7.3 Excel 导出

主结果导出仍然沿用：

- `对账结果`

差异导出仍然沿用：

- `聚天下有、第三方当月无`
- `第三方有、聚天下当月无`

导出结构本身不需要升级。

## 8. 错误处理

同程接入后至少需要覆盖以下错误场景：

- 平台文件不是单工作表
- 平台文件缺少必要字段
- `旅游日期` 无法解析
- 聚天下文件缺少 `订单列表`
- 聚天下文件缺少 `订单号`

建议错误提示风格延续当前项目面向业务用户的表达，例如：

- `同程文件必须且只能包含一个工作表，当前检测到 2 个`
- `同程文件缺少必要字段: 三方流水号`
- `同程文件中的“旅游日期”存在无法识别的日期，请检查原始数据格式`

如果目标月份过滤后无有效记录，则不应报系统错误，而应返回：

- 空主结果表
- 正常差异结果
- 正常摘要统计

## 10. 已识别的实现风险

当前仓库里的 `test_data` 已被切换为同程样例，因此旧的真实样例集成测试存在客观失效风险：

- `tests/integration/test_douyin_reconciliation.py`
- `tests/integration/test_meituan_reconciliation.py`

它们失败的原因不是功能回归，而是所依赖的样例文件已不在仓库中。

因此，同程接入实施时需要同步处理测试基线，至少做到：

- 保留现有单元测试和 Web 测试绿色
- 让全量测试不再依赖已删除样例文件

## 11. 本次设计结论

本次同程扩展建议采用“共享读取层小幅补强 + 新增同程适配器 + 注册同程平台元数据和报表定义”的方案。

原因如下：

- 同程业务口径简单，完全兼容当前领域模型
- 现有页面和导出机制可直接复用
- 唯一需要额外处理的是动态单工作表读取
- 该能力一旦补齐，后续接入类似平台也可以直接复用

设计确认后，下一步应进入实施计划编写，并按测试优先顺序推进，不直接跳入编码。
